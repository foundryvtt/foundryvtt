Following the stable release of Foundry VTT Version 13, our sights are now set on the prioritization and planning stages of Version 14 and all the fun and excitement that comes with a new prototyping phase. While we will, of course, continue to release updates for V13 to maintain its stability and correct bugs, we've already begun laying out our plans for the next stage of Foundry VTT development. As always, that process begins with opening a poll for our Patreon Community to vote on a major feature which we will prioritize for development.

Planned Features

Of course, while our Patreon feature prioritization poll helps us select a feature the community would like us to prioritize, we also have our own internal goals planned for the forthcoming V14.

Feature Prioritization Poll

In addition to the features we're planning for our own development goals, we will prioritize one feature from the following list as selectted by our Patreon Supporters. It's important to emphasize that all of the features on this list are ones which we plan to implement at some future time; these votes are intended to help us choose which feature we'll develop right now, in this version, not choose the only feature from the list we will ever develop. It's also worthy of note that some of the options on these polls come and go depending on a number of factors, so we may have some options returning to this poll that weren't on the previous one, and some options that were on the previous poll may not be here. Options that are missing are not removed from our plans, merely not something we feel we could reasonably develop as part of the focus of this version.

Enough caveats and addendums and preamble, let's get to the choices:

Advanced Measured Templates

An expansion of the existing Measured Templates system and add new features like template animation support, improved customization for template shapes, and the ability to link Templates to a Token.

If you've ever wanted to visually see the arc of your lightning or the blast of a fireball, animations are for you. For the more defensively minded, linking a template lets you walk around with auras. Combat can be not only more exciting but sped up with accurate template-zones based on walls, allowing you to easily target (and blow up) the right people.

This feature would:

  1. Support attaching a Template to a Token so that the Template moves along with the Token.
  2. Improve integration with the targeting system so the placement of Templates can auto-target Tokens within its area of effect.
  3. Add an API framework for rendering beautiful animations inside of Template shapes.
  4. Add an option that will constrain the area of Measured Templates according to nearby walls.
  5. Add support for video textures to be used for Template fill textures.

This feature might:

  1. Include built-in animation options for common effects like Fire, ice, lightning, force-fields, and more.
  2. Add a framework to apply a visual filter or animation onto Tokens inside the Template area.
  3. Other improvements related to Measured Templates and their interaction with other software components.

Document Tags and Improved Search

You can never have enough tools in the toolbox when it comes to keeping your games organized. Document Tags would allow users to assign special Tags to any existing Document (such as Journal Entries, Actors, Items, etc.) solely for the purposes of improving organization and search-ability.

This feature would:

  1. Provide a convenient and easily-accessible way to search all available Documents by name or Tag.
  2. Allow any Document to be assigned Tags which map to a user-defined or package-defined hierarchy. For example:
    • an Actor could be given Tags like ancestry: elf, class: fighter:battlemaster, gender: non-binary, build: athletic.
    • An Item might be given Tags like: category: staves:magical, effect: spellcasting.
  3. A JournalEntryPage could be given Tags like location: town:ferncombe, organization:delvers, disposition: hostile.

This feature might:

  1. Allow using the search results for Document Tags to create a convenient Rollable Table for easy randomization, such as generating a table of enemies for a random encounter or randomized loot results.

Improved Drawing Tools

Sometimes your players just insist on going to places you don't have a scene prepared for. A lot of GMs fall back on using drawing tools to provide a rudimentary representation of a tactical map, and there's no harm in that! Many users have suggested that the drawing tools, while valuable as they exist now, could do with a little development love and some improved features.

This feature would:

  1. Provide a more streamlined interface for the existing drawing tools using a new application for quickly changing line thickness, fill, color settings, opacity, and degree of smoothing (for the freehand drawing tool).
  2. Provide ability to quickly switch between recently used colors using a color palette.
  3. Allow for additional quick-change options on the Drawings HUD, such as ability to quickly flip and rotate drawings.
  4. Include new line types such as dashed lines, and arrows.
  5. Improve the existing Text drawing type to support additional text styles and multiline editing.

This feature might:

  1. Provide enhanced features for freehand drawings, including an eraser tool, and potentially allowing multiple freehand strokes to combine into a single drawing Document.
  2. Provide a quick configuration for drop shadow on drawings, to help them stand out from the background or provide a simple sense of depth of a drawn object such as a wall.

Improved Vision and Detection Modes

The initial implementation of Vision and Detection modes was a great first step, but there are definitely some areas where we can improve upon that iteration. Prioritizing this feature would allow us to make the handling of vision modes a little more convenient for GMs and players alike in an effort to get that scene feeling as immersive as possible.

This feature would:

  1. Provide a way for the Owner of a Token to quickly toggle their active Vision Mode from the Token HUD.
  2. Support configuration of Vision and Detection Mode interaction with darkness sources.
  3. Allow configuring "Important" detection modes, which always render an effect on a detected token even if the token was detected by some other detection mode.

  4. Provide support for "Imprecise" detection modes, which only reveal the size and location of the detection token, but not the token's identity.
  5. Include an API for "can Token A see Token B (with Detection Mode X), at all, precisely, or imprecisely."

  6. Support detection modes, that on its own cannot detect anything, but make it possible to detect something that another detection mode couldn't normally. For example, See Invisibility would only detect an invisible Token if some other sight-based detection mode of the Token like Light Perception or Darkvision could see the Token if it didn't have the Invisible status.
  7. Remove Vision Range in favor of selecting the Detection Mode that the Token should view the canvas with.

This feature might:

  1. Support for grid-based point source shapes and range tests.
  2. Add additional vision modes effects to the existing list, such as effects to represent "Thermal" vision or "Echolocation".
  3. Improve the API for automatic detection mode preparation based on the actor senses.

Manual Fog of War

One of the most commonly requested features from fans of a more "classic" VTT experience. The goal here would be for us to provide a suite of tools allowing Gamemasters to have a more hands-on, table-like experience for managing the reveal of Fog of War for players by marking areas of the map as revealed or obscured.

This feature will:

  1. Provide a GM with tools to draw geometric shapes (rectangle, ellipse, polygon) that remove or add fog of war for players.
  2. Allow for manual fog of war exploration to be shared across all players.

This feature might:

  • Allow for dynamic fog of war exploration to be shared across all players.
  • Support a mode which prevents tokens from moving outside the explored fog of war area.
  • Support a hybrid fog of war experience where manually revealed fog of war is carried over to the dynamic fog of war system.

    • Special Effect Regions

    While ambient light sources offer a wide variety of animated effects that can really bring a scene to life. It's been a long-stated goal for us to provide a way for users to animate their scenes without having to also pierce the fog of war with a light. The goal for Special Effects Regions would be to build on the existing Scene Regions framework to allow GMs and content creators even more ways to make their scenes shine.

    This feature would:

    1. Extend the V12 Scene Regions system with a suite of Region Behaviors designed specifically to support visual effects rendered in scenes.
    2. Provide a set of built-in animated effects that can be applied to Scene Regions, such as:
      • Fire
      • Roiling Smoke
      • Swirling Fog
      • Rippling Water
      • Bubbling Lava
      • Electricity
      • And more...
    3. Provide an API framework allowing system and module developers to define new shader-based animations to be displayed on Scene Regions.

    This feature might:

    1. Support particle emitters in addition to shader-based effects.
    2. Support Ambient Audio effects within a specific region as an alternative to point-source based sound effects.
    3. Support configuring illumination options, allowing a GM to assign a region that pierces the fog of war.
    4. Support weather effects localized to a specific region.

    Canvas-Placeable Items

    Imagine the scene: your party enters a room dominated by a statue clutching a gleaming red gem, the legendary artifact they've been searching for. With placeable items, that legendary Item that has been stored in your Sidebar can now exist on the map for your players to loot directly into their inventory. If your players are motivated by exploration, discovery, and the pursuit of treasure - the Placeable Items system will allow you to provide deeper engagement and immersion for the environments you create.

    This feature effectively allows Items to be placed directly onto a Scene, unlocking the ability for Players to find and acquire treasure or plot-advancing items by interacting directly with the game Canvas.

    This feature would:

    1. Implement a new embedded Document type which allows Items to be placed within a Scene.
    2. Add a new Canvas layer that contains and displays Items which are present within a Scene.
    3. Placed Items can be viewed and added to Actor inventories.
    4. Placed Items can have a customized appearance similar to the features offered for Tokens.

    This feature might:

    1. Allow combining bundles of Items together which can support cases like a Treasure Chest that contains multiple things.
    2. Integrate placeable items with Detection Modes to allow certain items to be invisible or hidden from view and only reveal under certain conditions, for cases like a "detect magic" spell, or a "thermal goggles" effect.

    Pop-Out Application Windows

    Advanced support for multiple monitors has been something we've often wanted to pursue as a feature, and we now believe there is a reliable way for us to do so. Thanks to advancements made during V13's UX and UI overhaul, we hope that we could now provide a more reliable way for users to manage their interactions with Foundry VTT using multiple application windows.

    This feature would:

    1. Allow users to quickly move an application window out of the bounds of the Foundry VTT viewport, to a new browser tab.
    2. Provide better handling for users with multiple monitors, allowing them to dedicate secondary screen real-estate to particular Applications.

    This feature might:

    1. Remember the last-used position and state of a pop-out window, so that it is automatically restored to the previous position and size when it is re-opened.
    2. Allow Users to pop out specific UI control elements, such as the Audio Visual Chat window, hiding them from the main application window rather than merely duplicating them.
